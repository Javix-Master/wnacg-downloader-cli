# Reliability & Rate-Limit Patterns

WNACG 的主要操作風險是**被站點限速／封 IP**。以下是實測歸納的可靠下載模式。

## 並行與間隔
- 一般：`--concurrency 3`。
- 大本（>200–300 頁）或重試時：降到 `--concurrency 2`。
- 批量：務必 `--comic-interval 300`（每本間隔 5 分鐘），避免連續請求被視為攻擊。

## 429 / IP 被限速
- `download_image` 內建退避重試：429 等待 `5s × 次數`，逾時／5xx 等待 `1.5s × 次數`；以 `--retries` 調整次數（預設 3）。
- 即使有間隔與低並行，大本仍可能在中後段大量觸發 `IP 被限速`（曾觀察到 327 頁僅 65 張成功）。
  - 對策：先暫停，稍後以更低並行重試；不要硬跑到底。

## 部分成功的處理
- 下載完成會回報 `成功 N/total`，失敗張數列在 stderr。
- 只有**零張成功**才算整本失敗（exit 非 0）；部分成功仍保留已下載內容。
- 補齊：對同一本加 `--force` 重跑（已存在的圖片會略過，只補缺的）。
- 大本部分成功（如 253/332）仍有價值，先保留再決定是否重試。

## 暫存與中斷安全
- 下載過程寫入 `<目錄>/.下载中-<標題>/`，全部完成才改名為正式目錄。
- 中斷或失敗後可安全重跑；殘留的 `.下载中-*` 目錄可手動清除。

## 站點「下載漫畫」按鈕 / Server 2 直連（選用備援）
- 站點頁面的「下載漫畫」按鈕會導向打包 ZIP（含 Server 2 備用線路）。
- 實測：該直連連結時效短、常見 403，且檔名未正規化。
- 結論：僅作為高頁數本子的一次性測試，**不要依賴**；以 CLI 的逐圖下載為主路徑。

## Agent / 背景執行
- 加 `--no-progress` 讓 log 乾淨。
- 背景批量完成後，立即檢查下載目錄、清點各本圖片數、處理失敗項，再進行下一批。
- 以 exit code 與 stderr 判斷成敗，不要假設成功。
